执行流程

分成了server层和存储层

1.连接器：

连接方式有TCP/IP、命名管道或共享内存、UNIX域套接字

创建连接的时候会创建一条线程来是让服务端和客户端交互

调开连接的时候服务端会缓存这条连接给下一个客户端使用，避免了频繁创建销毁线程

使用长连接减少重连的次数，但是定期重新初始化资源，否则占用内存会过大

2.查询缓存：

查询缓存可以在不同客户端共享

如果查询语句有字符不同或者调用了系统函数如NOW()，不会存放到缓存

表结构或者表数据被修改的时候，缓存失效

3.分析器：查询sql语句是否符合语法

4.优化器：

选择什么索引

表之间的连接顺序

5.执行器（存储引擎）：对表中的数据进行查询或者写入

日志系统

undo log（回滚日志）

为什么需要undo log

undo log是 Innodb 存储引擎层生成的日志，实现了事务中的原子性，防止异常（进程崩溃、主机宕机）时数据错误，主要用于事务回滚和 MVCC。

undo log日志格式

  • INSERT操作对应的 undo log日志：根据聚簇索引记录一条undo log
    • end of record：地址
    • undo type：日志类型
    • undo no：日志编号
    • table id：表id
    • 主键各列信息
    • start of record：地址
  • DELETE操作对应的 undo log日志：根据聚簇索引记录一条undo log
    • delte mark：逻辑删除，delete_flag设为1，但是还在正常记录的单向链表
    • purge：事务提交后，真正删除，把记录移动到垃圾链表中
  • UPDATE操作对应的 undo log日志：
    • 不更新主键
      • 就地更新：所有被更新的列更新前后对比，占用的存储空间一样大
      • 先真正删除旧纪录，再添加新纪录
    • 更新主键：记录两条undo log
      • 先对旧纪录delete mark
      • 再创建新纪录插入到聚簇索引中，说过了，树就是索引
      • 如果有二级索引，还需要对二级索引进行上述操作

事务回滚

每当 InnoDB 引擎对一条记录进行操作（修改、删除、新增）时，要把回滚时需要的信息都记录到 undo log 里，然后反操作

insert和update\delete有两个不同的undo log 的格式，事务提交后，insert回滚段被删除，update回滚段还存在

在发生回滚时，就读取 undo log 里的数据，然后做原先相反操作。比如当 delete 一条记录时，undo log 中会把记录中的内容都记下来，然后执行回滚操作的时候，就读取 undo log 里的数据，然后进行 insert 操作

和redo log一样也有buffer，不过是放在buffer pool存储的undo页，先持久化undo log，再持久化数据，最后提交事务

MVCC

  • 通过 trx_id 可以知道该记录是被哪个事务修改的；
  • 通过 roll_pointer 指针可以将这些 undo log 串成一个链表，这个链表就被称为版本链；

这种通过「版本链」来控制并发事务访问同一个记录时的行为就叫 MVCC（多版本并发控制）

通过版本链找到视图可见的那个数据版本

快照读

SQL读取的数据是可见版本，也就是历史版本，普通的SELECT就是快照读。

使用MVCC保证不出现幻读

当前读

SQL读取的数据是最新版本。通过锁机制来保证读取的数据无法通过其他事务进行修改UPDATE、DELETE、INSERT、SELECT … LOCK IN SHARE MODE、SELECT … FOR UPDATE都是当前读。

通过neck key锁保证别的事务的修改会被阻塞避免幻读

任务视图

一个数据版本，对于一个事务视图来说，除了自己的更新总是可见以外，

查询语句还有三种情况： -

  1. 版本未提交，不可见；
  2. 版本已提交，但是是在视图创建后提交的，不可见；
  3. 版本已提交，而且是在视图创建前提交的，可见。

更新语句使用“当前读”，能够拿到最新版本的数据，select语句如果加锁也是当前读

如果其他事务修改了数据，产生最新版本，但是事务没有提交的话，当前事务需要修改数据就会等待行锁释放，查询则不需要

redo log（重做日志）

为什么需要redo log

redo log 是 Innodb 存储引擎层生成的日志，实现了事务中的持久性，用来记录事务中的sql语句，主要用于掉电等故障恢复，将丢失的脏页刷盘，故障后的恢复

redo log 是为了防止 Buffer Pool 中的脏页丢失而设计的

  • 保存数据量小，只有数据到了磁盘才算是数据不会丢失，但是我们不可能频繁的将脏页merge到磁盘，这是随机I/O，效率很低，所以就有了redo log，记录下事务中所有的更新操作，在合适的时候将buffer pool的脏页merge到磁盘
  • redo lo刷盘是顺序I/O，效率较高，但是为了不频繁的顺序I/O，就有了redo log buffer，一定条件下会将redo log buffer内的 redo log 持久化到磁盘

redo log格式（包括但不限于这些）

type：日志的类型

space ID：表空间ID

page number：页号

offest：页面中的偏移量，即被修改的记录的位置

data：具体修改了什么

事务和redo log的关系

一个事务包含很多sql语句，一个语句又包含很多MTR，一个MTR包含一组redo log

MTR(mini-transaction)可以理解为一组 redo log，比如插入操作会修改B+树，为了保证B+树的正确性，redo log以组分类，如果有多个索引，那么还需要有多组 redo log，一组redo log 是一个不可分割的整体

redo log buffer

是一片连续的物理空间，存放log block，log block就是一个用来存放redo log的页

每当产生一组 redo log 时，即一个MTR，会先写入到 redo log buffer，后续在持久化到磁盘

  • buffer_next_to_write：下一个需要刷盘的MTR
  • buffer_free：下一个MTR可以写入log buffer的位置

刷盘的位置？

flushed_to_disk_lsn 之后到 lsn 才会刷盘

什么时候刷盘？

  • MySQL 正常关闭时；
  • 当 redo log buffer 中记录的写入量大于 redo log buffer 内存空间的一半时，会触发落盘；
  • InnoDB 的后台线程每隔 1 秒，将 redo log buffer 持久化到磁盘。
  • 每次事务提交时都将缓存在 redo log buffer 里的 redo log 直接持久化到磁盘（这个策略可由 innodb_flush_log_at_trx_commit 参数控制，下面会说）。

binlog （归档日志）

是 Server 层生成的日志，主要用于数据备份和主从复制；

  • binlog 是追加写，写满一个文件，就创建一个新的文件继续写，不会覆盖以前的日志，保存的是全量的日志。
  • redo log 是循环写，日志空间大小是固定，全部写满就从头开始，保存未被刷入磁盘的脏页日志。

什么时候刷盘

事务执行过程中，先把日志写到 binlog cache（Server 层的 cache），事务提交的时候，再把 binlog cache 写到 binlog 文件中。

fsync才会涉及磁盘I/O

数据备份

一个线程只能有一个事务

一个事务的 binlog 是不能被拆开的，所以每当执行一个 begin/start transaction 的时候，就会默认提交上一个事务

主从复制

  • 写入 Binlog：主库写 binlog 日志，提交事务，并更新本地存储数据。
  • 同步 Binlog：把 binlog 复制到所有从库上，每个从库把 binlog 写到暂存日志中。
  • 回放 Binlog：回放 binlog，并更新存储引擎中的数据。

Buffer Pool

一个控制块对应一个缓冲页

通过表空间ID和页号为key进行哈希确定页存放的位置

  • free链表：记录空闲的缓冲页
  • flush链表：记录Buffer Pool的脏页，特定时间刷盘
  • LRU链表：当Buffer Pool满了，如何进行淘汰使用率不高的页，提高Buffer Pool命中率

简单的LRU链表：用到某个缓存页，就把它放到链表头部，淘汰只需删除链表尾部的缓存页

但是预读或者是全表扫描可能会添加使用率不高的很多页到Buffer Pool

两段式LRU链表：

young区存储热数据，old区存储冷数据，访问的缓存页先放在冷数据，如果在一定时间间隔外再次访问，就移动这个缓存页到young区，在young区内的缓存页再次被访问，如果在前1/4的young区，就不会移动，否则移动到链表头部

Undo 页是记录什么？

开启事务后，InnoDB 层更新记录前，首先要记录相应的 undo log，如果是更新操作，需要把被更新的列的旧值记下来，也就是要生成一条 undo log，undo log 会写入 Buffer Pool 中的 Undo 页面。

提高了读写效率，减少了从磁盘读取数据页来更新：

当读取时，如果数据在Buffer Pool中，可直接返回；

当修改数据时，如果数据在Buffer Pool中，可直接修改，否则从磁盘中获取再修改，再特定时间后刷盘；

两阶段提交

事务提交后，redo log 和 binlog 都要持久化到磁盘，但是这两个是独立的逻辑，可能出现半成功的状态，这样就造成两份日志之间的逻辑不一致。

只有redo log提交成功了，那么故障后，只会恢复主库的数据，从库根据binlog复制，出现主从数据不一致

只有binlog提交成功了，那么故障后，从库根据binlog复制，出现主从数据不一致

事务提交时会开启一个内部 XA 事务，对应一个XID

可以看到，对于处于 prepare 阶段的 redo log，即可以提交事务，也可以回滚事务，这取决于是否能在 binlog 中查找到与 redo log 相同的 XID，如果有就提交事务，如果没有就回滚事务。这样就可以保证 redo log 和 binlog 这两份日志的一致性了。

事务

什么是事务

事务是保证ACID对一个或多个数据库整体的操作

事务是很多sql语句的总和，它确保了这些sql语句能够一起得到正确的执行，在发生错误时能够撤销事务中的sql语句对数据的操作

如何开启一个事务

  • 隐式开启：执行一条sql语句的时候，会自动开启事务，由系统变量autocomit控制

  • 手动开启

    • 执行了 begin/start transaction 命令后，并不代表事务启动了。只有在执行这个命令后，执行了增删查改操作的 SQL 语句，才是事务真正启动的时机；

    • 执行了 start transaction with consistent snapshot 命令，启动一致性读，就会马上启动事务。

如何提交一个事务

  • 自动提交：默认是自动提交事务
  • 手动提交：在需要多条sql语句共同执行时，关闭事务自动提交，commit提交或者rollback回滚
  • 隐式提交：在开启下一个事务之前，会自动将上一个事务提交、或者使用DDL语句也会将上一个事务提交

事务的状态

  • 活动的：正常执行的
  • 部分提交的：还没刷盘
  • 失败的：执行出错或者故障或者刷盘失败
  • 中止的：回滚的
  • 提交的：

ACID（即原子性、一致性、隔离性、持久性）

  • 原子性（使用undo log）：一个事务中的所有操作都是不可分割的，要么全部完成，要么全部失败
  • 隔离性（锁 + MVCC）：防止多个事务间并行操作时，其它事务对本事务操作数据的影响
  • 持久性（使用redo log）：事务提交后，数据被保存在磁盘中，数据不会丢失
  • 一致性（所追求）：结果符合约束，比如有钱才能转账、转账双方加起来银行账户的钱总和总是不变

隔离性与隔离级别

隔离性可能问题：脏读、不可重复读、幻读

脏读

事务A查询到事务B未提交的修改数据

不可重复读

事务B提交了修改的事务后，事务A前后两次查询数据不一致

幻读

同一条语句在事务中前后查询的记录不一致

数据一致性问题，也包括了日志的一致性问题，日志是根据事务提交时间写入的，事务提交的晚可能就会修改其他事务的结果 ，加锁可以阻塞事务的提交，即使把所有的记录都加上锁，还是阻止不了新插入的记录，*记录数量不一样 *

快照读不会加锁，当前读会加锁

隔离级别：读未提交、 读提交、可重复读和串行化

  1. 读未提交：事务A未提交事务，在事务A中能看到 事务B 未提交事务时修改的数据
  2. 读提交：事务A未提交事务，在事务A中能看到 事务B 提交事务后发生变动的数据，可能在事务A中不希望能看到在操作时数据被修改
  3. 可重复读：事务A未提交事务，数据不受事务B的修改操作的影响，数据前后一致性
  4. 串行化：事务A未提交之前，事务B不允许修改数据，需要等待事务A提交事务

每条记录在更新的时候都会同时记录一条回滚操作，记录最新值

系统里没有比这个回滚日志read-view更早的read-view的时候，回滚日志将被删除，事务提交可能就能删除回滚日志，所以要避免使用长事务，占用内存过多，怎么避免？显示启动事务，因为可能有客户端会将自动提交事务关闭

加锁规则

  • 原则1：加锁的基本单位是next-key lock。希望你还记得，next-key lock是前开后闭区间。
  • 原则2：查找过程中访问到的对象才会加锁。
  • 优化1：索引上的等值查询，给唯一索引加锁的时候，next-key lock退化为行锁。
  • 优化2：索引上的等值查询，向右遍历时且最后一个值不满足等值条件的时候，next-key lock 退化为间隙锁。
  • 一个bug：唯一索引上的范围查询会访问到不满足条件的第一个值为止。

全局锁：会话断开能主动释放锁

整个库会锁住，无法进行更新、删除操作，也没办法修改字段，主从表数据备份也会出问题

如果是引擎支持可重复读的隔离级别，可在备份前开启事务，在备份途中一直是同一个视图

否则需要加锁，备份的时候需要一致性视图，所以要使用FTWRL命令加锁，为什么不用自带的mysqldump，因为可能引擎不支持事务

表级锁：读写互斥

表锁： lock tables …read/write，会话断开能主动释放锁

元数据锁MDL：读写锁间互斥，事务提交才释放

当对一个表做增删改查操作的时候，加MDL读锁；当要对表做结构变更操作的时候，加MDL写锁。

读锁之间不互斥，因此你可以有多个线程同时对一张表增删改查。

读写锁之间、写锁之间是互斥的，用来保证变更表结构操作的安全性。因此，如果有两个线程要同时给一个表加字段，其中一个要等另一个执行完才能开始执行。

事务中的MDL锁，在语句执行开始时申请，但是语句结束后并不会马上释放，而会等到整个事务提交后再释放，事务不提交，就会一直占着MDL锁，如果占了写锁，导致表就不可读写，后续操作失效，并且重试机制可能导致库的线程爆满，所以要避免长事务

意向锁

  • 在使用 InnoDB 引擎的表里对某些记录加上「共享锁」之前，需要先在表级别加上一个「意向共享锁」；
  • 在使用 InnoDB 引擎的表里对某些纪录加上「独占锁」之前，需要先在表级别加上一个「意向独占锁」；

有了意向锁，那么加独占表锁的时候就可以不用遍历，如果有记录存在独占锁，可以直接添加

意向锁和共享表锁、独占表锁lock tables …read/write是读写互斥的

AUTO-INC 锁：执行完插入语句后就会立即释放

但是多事务并行情况下，效率较低

轻量级锁：一样也是在插入数据的时候，会为被 AUTO_INCREMENT 修饰的字段加上轻量级锁，然后给该字段赋值一个自增的值，就把这个轻量级锁释放了，而不需要等待整个插入语句执行完后才释放锁，如果同时binlog_format=statement，在「主从复制的场景」中会发生数据不一致的问题，两个session的insert是交错进行的，但是从库的拿到的binlog记录的sql语句不是交错的，是一个session接着一个session的，就会发生数据不一致，需要修改binlog_format = row

行锁：事务提交才释放，S共享锁和X独占锁互斥

在事务提交后才会释放行锁，所以有冲突的修改语句尽可能的放到事务的末尾执行，这样冲突的时间会尽可能的减少

唯一索引范围查询：

大于等于：如果等值存在的话，在等值的那点多一个退化的记录锁，加上分开的多个间隙锁加一个next-key锁

大于：分开的多个间隙锁加一个next-key锁

  • 当条件值的记录「不在」表中时，那么不管是「小于」还是「小于等于」条件的范围查询，扫描到终止范围查询的记录时，该记录的主键索引中的 next-key 锁会退化成间隙锁，就是要找到大于该查询条件的第一个值，在这个值的左边都要被锁住，其他扫描到的记录，都是在这些记录的主键索引上加 next-key 锁。
  • 当条件值的记录「在」表中时：
    • 如果是「小于」条件的范围查询，扫描到终止范围查询的记录时，该记录的主键索引中的 next-key 锁会退化成间隙锁，其他扫描到的记录，都是在这些记录的主键索引上，加 next-key 锁。
    • 如果是「小于等于」条件的范围查询，扫描到终止范围查询的记录时，该记录的主键索引中的 next-key 锁「不会」退化成间隙锁，其他扫描到的记录，都是在这些记录的主键索引上加 next-key 锁。

如果查到了一条记录，则先添加next-key 锁(n-1，n]，然后为该记录的主键添加记录锁，然后继续往后查询，找到第一个不符合的，左边的到第一个记录的都加锁；插入左括号或者右括号的数值可能可以成功，只要插入的记录的主键不在左右括号的记录的主键之间

连接

连接就是把各个表的记录拿出来依次匹配，结果返回给客户端

连接分类

  • 内连接：ON、WHERE作用一样
  • 外连接：把被驱动表不符合ON条件的记录补NULL加入到结果集

连接算法

循环嵌套连接：

  1. 先用代价最低的方式对驱动表单表查询
  2. 对1中驱动表的每一条都在被驱动表中查询匹配的记录
  3. 如果是多表连接查询，1、2步骤的结果集当作新的驱动表，重复上述步骤

基于块的循环嵌套连接：

  • 使用Join Buffer先缓存一部分驱动表数据，在被驱动表数据从磁盘中加载到内存和驱动表记录匹配的时候，不用加载那么多次，
  • 驱动表的每一条都在被驱动表中查询匹配的记录，如果先缓存了10条驱动表数据，那么I/O次数可能就会是原来的1/10

查询的成本

  • I/O成本：磁盘加载到内存的时间成本，读取一页的成本默认为1.0
  • CPU成本：读取以及检测一个记录是否符号和条件的成本为0.2

索引统计数据的成本：IN条件中的单点区域过多，就会使用索引统计数据粗略计算，估算索引列单个值的重复次数，然后估算回表记录数

单表查询成本

单表连接查询总成本 = I/O成本 + CPU成本

查询可能用到的索引

计算全表扫描的代价：

  • I/O成本：聚簇索引占用的页面数 * 1.0
  • CPU成本：单表总记录数 * 0.2

计算不同索引扫描的代价：

  • I/O成本：索引扫描区间 * 1.0 + 回表的记录数 * 1.0
  • CPU成本：读取索引的记录数 * 0.2 + 回表的记录数 * 0.2

选取代价最低的执行查询

多表查询成本

条件过滤：猜测驱动表的扇出值

多表连接查询总成本 = 单次访问驱动表的成本 + 驱动表扇出值 * 单次访问被驱动表的成本

索引是为了加快获取数据

MySQL 的基本存储结构是页

内节点（非叶子节点）存放目录项记录，主键索引存放的是主键值和页号，二级索引存放的是索引值，主键值和页号

二级索引可以理解为一个联合索引

MYISAM的索引都是二级索引，行号才是真正的主键索引，他存放的时候没有按照主键索引的顺序存放，数据和索引是分开的两种东西

  • 数据页和数据页之间，组成一个双向链表；

  • 每个数据页中的记录，是一个单向链表；

  • 每个数据页都根据内部的记录生成一个页目录（Page directory），如果是主键的话，可以在页目录中使用二分法快速定位；

  • 如果我们根据一个非主键、非索引列进行查询，那么需要遍历双向链表，找到所在的页；再遍历页内的单向链表；如果表内数据很大的话，这样的查询就会很慢。

相关的数据结构：是哈希表、有序数组和搜索树

  1. 哈希表：适合单个查询

    优点：查询单个数据很快，适用于等值查询 O(1)

    缺点：因为插入时是无序的，查找某个数据区间的数据会很慢，需要查询整张表 O(N)

  2. 有序数组：适合静态数据，很少更新的

    优点：查询数据很快，二分查找；因为有序，范围查找也很快，二分查找找到一端，然后往后遍历 O(logN)

    缺点：更新很慢，插入或者删除要移动数组 O(N)

  3. 搜索树:

    优点：查询用二分查找 O(logN)，更新需要维持搜索树特性，也是O(logN)

    缺点：树的高度会影响数据块大小，影响查询磁盘的次数，而查询磁盘是比较慢的，也就升级未N叉树，树根的数据块存储在内存中

    为什么是B树，数据大多在磁盘，需要I/O，B-tree能在保证增删改查的效率下让I/O次数减少

    为什么是B+树，在B树的优点之外，只有叶子节点存放数据，查询效率稳定，都是查询树的高度；叶子节点之间有双向链表链接，范围查询方便

索引的类型

  • 聚簇索引：存储整行数据

索引列的值必须是唯一的，并且不能为空，一个表只能有一个聚簇索引。

  • 唯一索引：存储主键数据

索引列的值是唯一的，值可以为空。

  • 普通索引：存储主键数据

没有什么限制，允许在定义索引的列中插入重复值和空值。

  • 复合索引：存储索引内的字段

也叫组合索引，用户可以在多个列上组合建立索引，遵循“最左匹配原则”，在条件允许的情况下使用复合索引可以替代多个单列索引的使用。

只有条件是=号的时候才会用到联合索引

聚簇索引（只有一个）和非聚簇索引

非主键索引的查询需要多扫描一棵索引树，这个过程称为回表，因为非主键索引值没有保存整行数据

尽量使用主键查询

普通索引和唯一索引（键值对应的值是唯一的）

对于普通索引来说，查找到满足条件的第一个记录(5,500)后，需要查找下一个记录，直到碰到第一个不满足k=5条件的记录。

对于唯一索引来说，由于索引定义了唯一性，查找到第一个满足条件的记录后，就会停止继续检索。

如果这个数据页还没有在内存中

对于唯一索引来说，需要将数据页读入内存，判断到没有冲突，插入这个值，语句执行结束

对于普通索引来说，更新操作先记录在change buffer，语句执行结束，下次查询时，将change buffer中的操作应用到原数据页，得到最新结果的过程称为merge

对于写多读少的业务，change buffer能够很好的减少对磁盘的访问（不用先将数据提取到内存）

索引维护

B+树为了维护索引有序性，在插入新值的时候需要做必要的维护

合并和分裂

自增主键的插入数据模式不会触发叶子节点的分裂

主键长度越小，由于非主键索引存储主键数据，普通索引的叶子节点就越小，普通索引占用的空间也就越小

索引用来排序

全表扫描排序的话是在内存或者磁盘中进行，文件排序，效率较低，可以通过LMIIT降低数据量

如果排序用到了索引，已经排序好的，就可以直接返回了

覆盖索引优化

在使用非主键索引查询的时候，只需要查询主键，而非主键索引存储主键数据，就不用回表

联合索引：高频请求索引内的字段，使用覆盖索引，不需要回表

最左前缀原则

索引在B+树的位置是按索引字段的值排序的，比如查询‘张%’，会出现连续的很多张xx的一个区间

所以当已经有了(a,b)这个联合索引后，一般就不需要单独在a上建立索引了，但是可能再要维护一个b的索引，这时候，我们要考虑的原则就是空间了，尽量b < a

联合索引的最左匹配原则，在遇到范围查询（如 >、<）的时候，就会停止匹配，也就是范围查询的字段可以用到联合索引，但是在范围查询字段的后面的字段无法用到联合索引。注意，对于 >=、<=、BETWEEN、like 前缀匹配的范围查询，并不会停止匹配（下一个=）

字符串字段创建索引

  1. 直接创建完整索引，这样可能比较占用空间；
  2. 创建前缀索引，节省空间，但会增加查询扫描次数，需要查询索引字段的时候会回表查询，不能使用覆盖索引；
  3. 倒序存储，再创建前缀索引，用于绕过字符串本身前缀的区分度不够的问题；
  4. 创建hash字段索引，查询性能稳定，有额外的存储和计算消耗，跟第三种方式一样，都不支持范围扫描。

索引下推优化（联合索引）

查询条件中有索引的等值查询，索引内部先去除不符合索引字段的值的记录，减少回表次数

即使联合索引不是连续的情况，也可以用来减少回表的次数

索引失效

利用索引的前提是索引里的 key 是有序的

左模糊查询失效

因为B+树是根据字段的字典序来排序的，而左模糊用不到从左开始的字典序

函数失效

索引排序是根据字段原本的值来排序，不是函数计算后的值 where length(name) = 6

表达式计算失效

也是没有用到排序的本来数据的值 where id + 1 = 10

隐式类型转换失效

字符串和数字比较的时候，mysql会使用函数自动把字符串转为数字，然后再进行比较

也就是使用函数失效 where phone = 134564822，索引用了函数转换

联合索引非最左匹配失效

在联合索引的情况下，数据是按照索引第一列排序，第一列数据相同时才会按照第二列排序，先用a，再用b

索引是 (a，b，c) ，where a = 1 and b = 2 || where b = 1 and c = 2 后者没有用a，自然没办法用b，索引失效，但还是有索引下推的优化

OR语句失效

OR是两个条件取并集，在考虑一个条件的索引时候，另一个条件可能和这个索引无关，导致WHERE条件为TRUE，全局扫描

因为OR 的含义就是两个只要满足一个即可，只要有一个不是索引字段，自然就会全局扫描

隐式字符编码转换失效

utf8（uft8mb3） -> utf8mb4

where a = b，理解为a是字段， b是值，函数作用在a会有问题，b不会

缺点

  • 需要占用物理空间，数量越大，占用空间越大；
  • 会降低表的增删改的效率，因为每次增删改索引，B+ 树为了维护索引有序性，都需要进行动态维护

什么时候加索引

  • 能在尽可能少的磁盘的 I/O 操作中完成查询工作；
  • 数据量大、区分度大
  • 需要排序的，因为索引查询本来就是有序的
  • 要能高效地查询某一个记录，也要能高效地执行范围查找；

7）分库分表

单一数据库的容量不足应对海量数据的存储导致的查询性能差的问题

简单来说，分库是将原本的单库拆分为多个库，分表是将原来的单表拆分为多个表。

7.1）分库

1）垂直分库：根据业务拆分

优点：

  • 可以针对不同业务场景优化数据库，提高性能。
  • 提高了数据库的并发能力。

缺点：

  • 需要处理分布式事务的问题。
  • 增加了系统的复杂度。

2）水平分库：

通过某个字段将数据分散至多个库或表中

优点：

  • 提高了单库的读写性能，降低了单库数据量。
  • 可以将库存储在不同的物理服务器上，提高了查询效率。

缺点：

  • 需要处理跨库查询的问题。
  • 对分库规则的设计需要谨慎，避免热点数据集中在某个库中。

7.2）分表

1）垂直分片

优点：

  • 拆分后每个表的数据量变小，查询时涉及的磁盘 I/O 次数相对减少，提高了查询效率。
  • 每个小表的并发写入操作相对较少，减少了数据库锁的竞争，提高了并发能力。

缺点：

  • 需要在应用层面处理跨表查询的逻辑，增加了开发的复杂性。
  • 如果一个事务涉及多个小表，可能需要在应用层面进行事务管理，增加了代码的复杂性。
  • 需要额外的措施来保证拆分后的小表之间的数据一致性。

2）水平分片

优点：

  • 可以根据数据量的增长动态地增加分表，从而扩展数据库的存储能力。
  • 每个小表的数据量减少，可以提高查询速度，尤其是在频繁查询的场景下。
  • 拆分后，每个小表的并发写入操作相对减少，降低了数据库锁的竞争，提高了并发性能。

缺点：

  • 对业务存在一定限制，如果没有按照分片键查询，会造成读扩散问题。
  • 对分表规则的设计需要谨慎，避免热点数据集中在某个表中。

垂直：因为根据业务拆分，粒度变细，查询的时候效率高，但是多业务更新的时候可能涉及到事务控制

水平：根据字段拆分，可以提高数据库存储能力，查询速度，但是可能导致数据集中在某个库或表中

7.3）什么场景分库分表？

写：高并发写的时候

读：数据量过大的时候，存储、查询效率差
