数据类型

有哪些数据结构

String类型（intset和SDS）

存放的是字符串

value可以是数字类型的字符串

Hash类型（ziplist和hashtable）

存储的是键值对

外层value是一个(field,value)键值对形式的map

作用：存放对象

List类型（ziplist和linkedlist）

可以存储相同的字符串

LPUSH头插法

RPUSH尾插法

作用：模拟队列，栈

Set类型（intset和hashtable）

不能存储相同的字符串

作用：好友判断

SortedSet类型（ziplist和skiplist）

外层value是一个(value,score)键值对形式的map，score用来排序，还有排名，关键是一定范围

作用：成绩排序

数据结构底层实现

String：SDS作为字符串表示

  • 使用的空间用 len 变量保存，查询长度不用遍历查询，效率高，没用的空间用 free 变量保存
  • 增加字符串长度会先用 len 检查是否有足够空间，不会造成缓冲区溢出导致覆盖其他空间的字符串
  • 内存分配次数可能减少，有预分配机制和惰性释放没用的空间
  • 可以保存二进制数据，因为是用 len 记录长度，而不是用空字符判断字符串是否结束，这样用空格字符切割就不会有问题
  • 沿用了C语言的结尾是’/0’，这样可以调用一部分C语言函数，比如字符串拼接等

字典hashtable：类似HashMap

  • 有ht[0]和ht[1]两个数组，一般来说是在ht[0]进行增删改查，ht[1]是用来扩容的
  • 扩容条件和HashMap差不多，但是可以收缩
  • 哈希冲突和HashMap差不多，但是HashMap在冲突的链表节点超过8个就会转成红黑树存储
  • 扩容不是一次性完成的，是由子进程分多次完成

跳跃表skiplist：大部分情况下，效率和平衡二叉树一样

由虚拟头节点（存放跳跃表信息）和多个节点组成

每个节点的某一层高有指向后一节点的指针和对应的跨度，

每个节点都有排位和可以存储一个字符串对象，如果排位相同，按字符串的字典序排序

整数集合intset

使用的长度用len表示，每个整数占用的空间用一个encoding属性表示

扩容的时候，先按新的encoding属性分配空间，再将原数组扩容，最后根据大小排序添加新的元素

但是降级的时候不会改变encoding属性

压缩链表ziplist

压缩链表节点包含迁移节点的占用字节，存放的数据

连锁更新：添加节点或者删除节点时，后一个节点可能要将存放新的前一节点的长度的字段拓展为5个字节，如果节点本身就在250-253字节之间，又会影响下一节点，不过出现概率很低

缓存一致性

1.先更新数据库还是先更新缓存？

  • 先更新数据库，再更新缓存 （新的线程进来更新了新的数据库，后面缓存用的是旧的）

  • 先更新缓存，再更新数据库 （新的线程更新了新的缓存，后面数据库更新的是旧的）

    因为并发都会出现缓存，数据库 数据不一致的情况

2.不更新缓存，而是删除缓存中的数据。然后，到读取数据时，发现缓存中没了数据之后，再从数据库中读取数据，更新到缓存中。
3.先更新数据库还是先删除缓存？
  • 先更新数据库，删除缓存 （缓存不存在才会可能有问题，更新了旧的数据库到缓存）
  • 先删除缓存，再更新数据库 （新的线程查询了了旧的数据，更新了旧的缓存，导致下次查询不会查询数据库）

常见的缓存更新策略？

  • Cache Aside（旁路缓存）策略；（redis使用）

  • Read/Write Through（读穿 / 写穿）策略；

  • Write Back（写回）策略；

Cache Aside（旁路缓存）

为什么「先更新数据库再删除缓存」不会有数据不一致的问题？

由于写数据库和写缓存的时间不一样，所以先写数据库再删除缓存中间的间隔时间会少一点，出问题的可能性更小

Cache Aside 策略适合读多写少的场景，不适合写多的场景，因为当写入比较频繁时，缓存中的数据会被频繁地清理，这样会对缓存的命中率有一些影响。

Read/Write Through（读穿 / 写穿）策略

应用程序只和缓存交互，不再和数据库交互，而是由缓存和数据库交互

Read Through 策略

先查询缓存中数据是否存在，如果存在则直接返回，如果不存在，则由缓存组件负责从数据库查询数据，并将结果写入到缓存组件，最后缓存组件将数据返回给应用。

2、Write Through 策略

当有数据更新的时候，先查询要写入的数据在缓存中是否已经存在：

  • 如果缓存中数据已经存在，则更新缓存中的数据，并且由缓存组件同步更新到数据库中，然后缓存组件告知应用程序更新完成。
  • 如果缓存中数据不存在，直接更新数据库，然后返回；

Write Back（写回）策略

实际上，Write Back（写回）策略也不能应用到我们常用的数据库和缓存的场景中，因为 Redis 并没有异步更新数据库的功能。

Write Back 策略特别适合写多的场景，因为发生写操作的时候， 只需要更新缓存，就立马返回了

但是带来的问题是，数据不是强一致性的，而且会有数据丢失的风险，参考文件刷盘前宕机

秒杀

全局唯一id：时间戳（当前时间减去初始时间）加序列号（每秒产生2^32）

线程并发：事务加锁

解决方法：1.悲观锁（例如syncronized）适合更新数据 2.乐观锁（版本号）成功率低 适合改变数据

超卖，加锁乐观锁

一人一单，悲观锁，但spring管理的事务被包在锁内才会防止事务提交前锁被释放，才能避免导致问题

分布式：不同服务器是不同的jvm控制，产生多个锁监视器，简单的锁（上述的锁）失效

锁误删问题：本线程阻塞时间超过了key过期时间，锁超时释放后被其他线程拿到，本线程唤醒后释放锁时释放了别的线程的锁

解决方法：判断是否是自己的锁（可能有垃圾回收机制阻塞，所以要判断和释放两个操作原子性），可以用lua脚本保证原子性

分布式锁-redission

用hash存储，大key是锁的标识，小key是线程标识，用于判断锁是否是自己的，

lock获取锁，没有就创建（设置过期时间），有就判断是自己的话，加1（多一层业务）

unlock释放锁，没有就返回，有就判断是自己的话，就先减一，再判断是否为0（业务都完成），删除key

异步秒杀

功能分区，判断资格部分在redis，然后开启另外的线程利用阻塞队列去完成数据库的操作

持久化

AOF日志：记录的是写操作的命令

为什么先执行命令，再把数据写入日志呢？

避免额外的检查开销，不会有语法错误

不会阻塞当前写操作命令的执行

数据可能会丢失，还没来得及写入AOF文件就宕机了

可能阻塞其他操作，因为写入AOF文件和写命令是同一个线程

AOF 写回策略有几种？

3种

写一次，更新一次

1s更新一次

草给操作系统决定什么时候更新

先写到buffer，再特定时机写道Page cache，再由操作系统决定什么时候刷盘

AOF 日志过大，会触发什么机制？

AOF 重写机制，新的AOF文件覆盖旧的AOF文件，新的AOF文件只记录最新的那个记录

重写 AOF 日志的过程是怎样的？

是由后台子进程 *bgrewriteaof* 来完成的，重写过程中，主进程依然可以正常处理命令，

重写 AOF 日志过程中，如果主进程修改了已经存在 key-value，那么会发生写时复制，此时这个 key-value 数据在子进程的内存数据就跟主进程的内存数据不一致了，这时要怎么办呢？

  • 执行客户端发来的命令；
  • 将执行后的写命令追加到 「AOF 缓冲区」；
  • 将执行后的写命令追加到 「AOF 重写缓冲区」；

子进程完成 AOF 重写工作，会向主进程发送一条信号，主进程收到该信号后，会调用一个信号处理函数，该函数主要做以下工作：

  • 将 AOF 重写缓冲区中的所有内容追加到新的 AOF 的文件中，使得新旧两个 AOF 文件所保存的数据库状态一致；
  • 新的 AOF 的文件进行改名，覆盖现有的 AOF 文件。

RDB 快照：记录的是实际的数据

RDB 做快照时会阻塞线程吗？

  • 执行了 save 命令，就会在主线程生成 RDB 文件，由于和执行操作命令在同一个线程，所以如果写入 RDB 文件的时间太长，会阻塞主线程；
  • 执行了 bgsave 命令，会创建一个子进程来生成 RDB 文件，这样可以避免主线程的阻塞；

Redis 的快照是全量快照，也就是说每次执行快照，都是把内存中的「所有数据」都记录到磁盘中。所以执行快照是一个比较重的操作

RDB 在执行快照的时候，数据能修改吗？

fork() 创建子进程，此时子进程和父进程是共享同一片内存数据的

如果主线程修改数据，就会会发生写时复制

混合持久化

既保证了 Redis 重启速度，又降低数据丢失风险。

AOF 文件的前半部分是 RDB 格式的全量数据，后半部分是 AOF 格式的增量数据。

混合持久化优点：

  • 混合持久化结合了 RDB 和 AOF 持久化的优点，开头为 RDB 的格式，使得 Redis 可以更快的启动，同时结合 AOF 的优点，有减低了大量数据丢失的风险。

混合持久化缺点：

  • AOF 文件中添加了 RDB 格式的内容，使得 AOF 文件的可读性变得很差；
  • 兼容性差，如果开启混合持久化，那么此混合持久化 AOF 文件，就不能用在 Redis 4.0 之前版本了。
