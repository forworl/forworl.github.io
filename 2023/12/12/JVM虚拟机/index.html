JVM虚拟机

发表于2023-12-12|更新于2024-01-21|JVM

思维导图

平台无关性

运行代码的两种方式

编译：把高级语言 -> 机器语言，高级语言就是c、java这些，低级语言就是汇编语言、机器语言这些

解释：通过解释器直接执行，不用编译成机器语言，这样效率较低

java是编译还是解释型的？

编译分为两部分

  1. 前期编译：javac 把 .java -> .class
  2. 后期编译（解释 + JIT即时编译优化）：JVM 把 .class -> 机器语言（2进制）

正常的代码是解释执行的，一句一句翻译成机器语言

对于热点代码，JIT优化是把热点代码直接编译成机器语言，再次访问不用就可以不用再翻译了

JIT（即时编译器）优化

由于JIT，所以以下都是基于热点代码，并且是在运行期进行的优化

项目重启的时候可以先降低请求量来做预热（触发JIT优化），再逐步提高流量

逃逸分析

全局逃逸：对象作为方法的返回值或者赋值给了静态变量

参数逃逸：方法内调用另一方法，对象作为参数传递，但是不会发生全局逃逸

无逃逸：对象只在方法内使用

锁消除

和逃逸分析相关，如果说synchronized用了对象锁，而这个对象无逃逸，那么说明这个对象是线程安全的，可以锁消除

锁膨胀

标量替换&&栈上分配对象

标量替换：如果对象无逃逸，那么可以将 对象.属性 -> 常量

栈上分配对象：如果对象无逃逸，那么可以在栈上分配对象而不是堆中分配对象，可以减少GC

方法内联

方法内调用了一个小型的方法（代码少，如 return a + b;），那么将这个小方法替换为方法内代码

JVM主要组成部分

类加载器：将Class文件字节码内容加载到内存中

运行时数据区：JVM内存

执行引擎：将内存中的Class文件内容转换成机器码文件，再交由 CPU 去执行

本地接口：在操作系统执行过程中可能要调用本地方法来实现程序

JVM内存模型/结构

JDK1.7和JDK1.8区别

  • JDK1.7

    • 运行时数据区：堆、方法区（永久代）、虚拟机栈、本地方法栈、程序计数器
    • 本地内存：直接内存

  • JDK1.8

    • 运行时数据区：堆、虚拟机栈、本地方法栈、程序计数器
    • 本地内存：直接内存、方法区（元空间）

  • 线程私有：程序计数器、虚拟机栈、本地方法栈

  • 线程共享：堆、方法区、直接内存 (非运行时数据区的一部分)

线程私有（指令，通过指令操作数据）

程序计数器

字节码的行号指示器，改变程序计数器的值，让程序可以往下执行

然后在线程上下文切换的时候，保存下程序计数器和栈的信息，在线程获取到CPU时间片的时候，恢复任务执行

虚拟机栈

执行Java 的方法调用，比如 f1() 调用 f2() 的时候，将 f1() 压入栈，调用完弹出

栈帧组成

局部变量表：存放编译器可知的基本数据类型（int等）和引用数据类型（reference）

操作数栈：进行数学运算的时候，用来存放被操作数（a、b）和中间结果（a + b）

动态链接: 当 f1() 调用 f2() 的时候，将指向方法区运行时常量池的 f2() 的符号引用转化为 f2() 地址引用，f1() 就可以调用 f2()

方法返回地址

该区域有哪些异常？

  1. 如果采用固定大小的 Java 虚拟机栈，那每个线程的 Java 虚拟机栈容量可以在线程创建的时候独立选定。如果线程请求分配的栈容量超过 Java 虚拟机栈允许的最大容量，Java 虚拟机将会抛出一个 StackOverflowError 异常
  2. 如果 Java 虚拟机栈可以动态扩展，并且在尝试扩展的时候无法申请到足够的内存；或者在创建新的线程时没有足够的内存去创建对应的虚拟机栈，那 Java 虚拟机将会抛出一个OutOfMemoryError异常

本地方法栈

虚拟机栈为虚拟机执行 Java 方法 （也就是字节码）服务，而本地方法栈则为虚拟机使用到的 Native 方法服务

特点和虚拟机栈类似，也有栈帧，也会抛异常，栈深度溢出StackOverflowError 和栈扩展失败OutOfMemoryError

线程共享（数据，类和对象）

堆

几乎所有的对象都在此分配，但是如果方法内的对象引用没有逃逸出方法，即没有被返回或者未被外面静态变量使用，就在虚拟机栈分配

组成

  • 新生代（伊甸园、两个幸存区）

  • 老年代

  • 永久代/元空间（逻辑在堆中，实际在本地内存中）

字符串常量池

利用缓存的思想，避免字符串的重复创建 ，底层是个HashMap

  1. 栈中引用直接指向串池中的字符串
  2. 栈中引用指向堆中字符串对象，这个对象再指向串池中的字符串

方法区

  1. 方法区是一种规范，具体实现有jdk1.7的永久代（在堆内存中），jdk1.8的元空间（在本地内存中）
  2. 存储Class文件类信息（方法、属性、构造器）、常量、静态变量、即时编译器编译后的代码缓存等数据
  3. 通过反射获取到的类型、方法名、字段名称、访问修饰符等信息就是从方法区获取到的

运行时常量池

存放Class文件编译期生成的各种字面量、符号引用，这些内容在类加载到内存后加入到运行时常量池中

  • 字面量：整数、浮点数、字符串字面量和 final 修饰的常量值
  • 符号引用：类符号引用、字段符号引用、方法符号引用、接口方法符号，引用的目标不一定加载到了虚拟机之中

直接内存（堆外内存）

直接内存并不是虚拟机运行时数据区的一部分

操作系统分配的内存，读写效率低

释放是通过unsafe.freeeMemeory()虚引用释放内存

对象解密

对象的创建过程

  • 类加载检查：是否可以在常量池中定位到这个类的符号引用（是否已经被解析），类是否被加载，没有的话加载类
  • 内存分配（实例化对象）
    • 分配方式
      • 指针碰撞：Serial、ParNew，适合没有内存碎片的
      • 空闲列表：CMS，适合有内存碎片的，会有一个列表维护空闲的空间
    • 分配时并发问题
      • CAS+失败重试：保证更新操作的原子性
      • TLAB（本地线程分配缓冲）：先在eden分配一块空间，尝试将对象分配到这块空间
  • 初始化对象
    • 初始化零值：给对象的实例字段分配默认值，避免使用的时候出现null
    • 设置对象头：类的元数据信息、hashCode、GC分代年龄、锁状态标志等
    • 执行 init 方法：把对象按照程序员的意愿进行初始化，字段赋值，一些前置的日志打印等
  • 引用指向创建的对象，分为直接引用和句柄引用

对象的结构

  • 对象头：

    • Mark Word：存储对象自身运行时的数据，如哈希码、GC分代年龄、锁状态标志
    • 类型指针：知道是哪个类的实例

  • 实例数据：存储Java类中各个类型的字段，包括基本类型和引用类型

  • 对齐填充：无意义，对象分配的内存是 8byte 的整数倍

如何访问对象

  • 句柄访问：reference指向包含对象实例数据、类型数据的指针，需要二次访问真正的数据，但是GC之后不用修改reference本身
  • 直接访问：reference直接指向对象，速度快

对象在堆中内存分配

对象优先在Eden分配，当eden没有足够内存，发动一次Minor GC（标记-赋值算法，把Eden和From的可达对象移到To）

大对象直接进入老年代，比如字符串、数组等

每次minor GC存活下来的对象年龄加1，长期存活的对象进入老年代、如果Survivor的相同年龄的对象占用内存大小达到Survivor的内存大小的一半，那么大于等于这个年龄的对象可以直接进入老年代，无需等到年龄15

对象担保策略

  1. 在Minor GC之前，会先检查老年代最大的连续内存空间是否大于新生代所有对象的内存空间，如果大于就认为此次Minor GC是安全的，否则认为有风险
    1. JDK6以前再次检查如果老年代最大的连续内存空间大于历史晋升到老年代的对象的占用空间平均值，如果大于那么会尝试一次Major GC，失败了才进行Full GC，如果小于直接进行Full GC
    2. JDK6以后再次检查如果老年代最大的连续内存空间大于历史晋升到老年代的对象的占用空间平均值，如果大于先尝试一次Minor GC，失败了才进行Full GC，如果小于直接进行Full GC

OOM异常原因

堆溢出

  • 内存泄露
  • 内存溢出：某些对象生命周期过长

栈溢出

  • StackOverflowError：栈容量固定大小，栈帧太大或者栈容量太小
  • OutOfMemoryError：栈容量动态拓展，不断创建线程，每个线程分配的栈内存大

方法区和运行时常量池溢出

CGLib字节码技术增强和动态语言，归结为类的内存溢出

本机直接内存溢出

默认大小和堆大小一样，直接用了Unsafe类分配内存，信息较少

垃圾回收

哪些东西是可以被回收的

  • 死亡的对象
  • 废弃的常量
  • 废弃的类

死亡对象判断方法

  • 引用计数法：对象被引用，计数器加1，缺点是难以解决循环引用的问题
  • 可达性分析法：从GC Roots开始，沿着引用链能到达的节点才算可达，Object6就不可达

哪些对象可以作为 GC Roots 呢？

哪些对象是不会被垃圾回收的

  • 在虚拟机栈(栈帧中的本地变量表)中引用的对象,譬如各个线程被调用的方法堆栈中使用到的
    参数、局部变量、临时变量等。
  • 在方法区中类静态属性引用的对象,譬如Java类的引用类型静态变量。
  • 在方法区中常量引用的对象,譬如字符串常量池(String Table)里的引用。
  • 在本地方法栈中JNI(即通常所说的Native方法)引用的对象。
  • Java虚拟机内部的引用,如基本数据类型对应的Class对象,一些常驻的异常对象(比如
    NullPointExcepiton、OutOfM emoryError)等,还有系统类加载器。
  • 所有被同步锁(synchronized关键字)持有的对象。

跨代引用问题

CMS和G1为了避免跨代引用导致扫描整个老年代，引入记忆集和卡表

记忆集Remembered Set和卡表CardTable

并发标记阶段使用

卡表是记忆集的具体实现方式，是一个数组，类似HashMap实现了Map

每个元素都是一个card（entry），key代表某一区域是否有跨代引用，value代表老年代

如果key为1，引用链就需要添加value代表的老年代的对象

写屏障

类似AOP的方式在更新对象引用前或后维护卡表

  • CMS 收集器在更新对象引用之后维护卡表
  • G1 收集器在对象引用前后都要维护卡表

三色标记

最终标记阶段使用

防止并发扫描的时候用户线程更改引用链的引用关系导致对象被误删

灰色在白色和黑色之间

  • 白色：垃圾收集器没有扫描到的节点，需要被清除
  • 黑色：垃圾收集器已经扫描到的节点，并且所有该黑色节点的引用也被扫描到
  • 灰色：垃圾收集器已经扫描到的节点，至少一个该灰色节点的引用没有被扫描到

对象消失

建立黑色和该白色的引用关系，同时将灰色到白色之间引用删除了

解决方案

增量更新（Incremental Update）：记录这些黑色建立白色引用关系，并发扫描后，重新扫描这些黑色，这些黑色就会转换成灰色，白色就转换成灰色

原始快照（SATB）：记录被删掉的原来这些灰色和白色引用关系，并发扫描后，重新按照原来的视图扫描一次

如何判断一个常量是废弃常量？

常量池的常量没有被任何对象引用，比如字符串常量池的常量”abc”没有被对象引用

如何判断一个类是无用的类？

  • 该类所有对象实例都被回收
  • 该类的类加载器ClassLoader被回收
  • 该类的类对象没有被引用，比如没有用反射方法

引用类型

  • 强引用：new出来的对象都是，永远不会回收
  • 软引用：内存不足了才回收，threadLocal的key就是软引用，可能造成内存泄露
  • 弱引用：比软引用对象生命周期更短，垃圾回收器线程扫描到就回收
  • 虚引用：任何时候都可能被回收，虚引用主要用来跟踪对象被垃圾回收的活动

垃圾回收算法

标记-清除算法

有大量内存碎片问题

标记-复制算法

能用的内存只有原来的一半，并且复制需要时间

标记-整理算法

也需要移动对象，效率不高

分代收集算法

  • 新生代：大部分对象生命周期短，需要复制的对象少，适合用标记-复制算法
  • 老年代：对象存活概率高，需要清除的对象不多，适合用标记-整理算法

垃圾回收器

都用到分代回收算法，不过后来的收集器越来越关心用户体验（收集过程的暂停时间）

  • Serial 收集器：只有一条垃圾回收线程，并且回收的时候会 stop the world

  • ParNew 收集器：Serial 收集器的多线程版本，能配合 CMS 回收器使用

  • Parallel 收集器：和ParNew 收集器差不多，多提供了很多参数供用户找到最合适的停顿时间或最大吞吐量

  • CMS 收集器：实现了用户线程和垃圾回收线程并发运行

    • 初始标记（stop the world）：先标记所有直接和 GC Roots 相连的二级节点
    • 并发标记：去标记引用链的其他所有节点
    • 重新标记（stop the world）：再标记上一步过程中用户线程新修改引用的节点
    • 并发清除：清除没有标记的节点

    • CMS 缺点
      • 由于是并发清除，无法处理浮动垃圾
      • 因为采用标记-清除算法，有大量内存碎片
      • 只能作用在老年代
    • CMS 优点
      • 耗时最长的并发标记和并发清除不用暂停所有线程，暂停时间STW短

  • G1收集器：主要针对配备多颗处理器及大容量内存的机器，内存的回收是以region作为基本单位的

    • 初始标记（stop the world忽略不计）：标记所有直接和 GC Roots 相连的二级节点，并修改 TAMS 指针的值
    • 并发标记：去标记引用链的其他所有节点，并且标记完后，对变动的节点标记
    • 最终标记（stop the world）：对变动的节点重新标记
    • 筛选回收（stop the world）：根据每个Region的价值选择回收顺序，并且将回收的Region中的剩下对象移动到空的Region

    • G1优点

      • G1清除的时候会STW，不会产生浮动垃圾

      • 老年代采用标记-整理算法，无内存碎片，新生代采用标记-复制算法

      • 作用整个堆，根据价值回收部分区域，减少暂停时间

    • G1缺点

  • GZC收集器：待办

类加载

.java -> 编译后变成.class -> 变成机器码再运行

Class文件结构

class文件是16进制的，可以在notepad++ 添加 Hex-Edictor插件打开

反编译class文件，到class文件所在目录使用javap命令，或者github下载jclasslib

  • 基本信息

    • 魔数：开头4个字节CAFEBABE区分是class文件，计算机是不以后缀名区分文件的
    • 版本号：class文件和jdk版本号要兼容

  • 常量池

    • 是一个数组，保存字面量和符号引用，字面量就比如字符串值或者定义为final类型的常量的值，符号引用就比如字段或者方法的名称
    • 保存了类的字段或者方法或者字段的值对应的字符串字面量在字符串常量池的索引，就比如abc = “abc”，那么在常量池会有一个位置保存abc，一个位置保存”abc”，这样可以区分是字段还是字段的值，但是都是指向相同的字符串字面量

  • 字段

  • 方法：将代码转换成一行行的字节码指令，和虚拟机栈的操作数栈和本地变量表配合，执行代码

  • 属性

类加载过程

加载、验证、准备、解析、初始化

  • 加载：

    • 获取类的二进制字节流，在堆中生成Class对象，Class对象是访问方法区的类的元信息的外部接口（反射的底层原理）
    • 在这里是程序员可控度最高的部分，可以自定义类加载器来实现类的加载，但是数组类型有些许不同

  • 连接

    • 验证：Class文件的字节流是否符合JVM规范
    • 准备：给static修饰的变量分配内存同时赋初值（一般是0）
    • 解析：将运行时常量池内的符号引用转换为直接引用（知道类的内存地址、知道方法的实际地址）

  • 初始化：构造函数使用（执行静态代码块，父类一定在子类前执行），给static变量赋值

类加载器

类加载器是负责加载类（在内存中生成一个Class对象），每个类都有一个引用指向它的类加载器

三个重要的类加载器

  • BootstrapClassLoader：启动类加载器，C++实现，获取的结果是null
  • ExtensionClassLoader：拓展类加载器
  • AppClassLoader：应用类加载器

自定义类加载器

  • 如果不想打破双亲委派机制，就重写findClass()
  • 如果想打破双亲委派机制，就重写loadClass()

双亲委派模型

只有父加载器反馈无法加载，才会让子加载器自己加载

目的是尽量让最顶层的BootstrapClassLoader来创建当前的类

双亲委派模型的好处

避免类的重复加载（从同一个Class文件加载，类名相同，但是类的类加载器不同也是不同的类）

打破双亲委派机制

重写loadClass()打破双亲委派机制

  1. 先检查类是否被加载过
  2. 调用父加载器加载
  3. 调用自己的加载器加载，可以通过重写findClass()自定义类加载器

运行期优化

前提是热点代码

逃逸分析：对象不会逃逸出方法

方法内联：方法较短，直接把方法体的代码放到调用者调用该方法的位置

字段优化：如果没有使用方法内联，并且需要频繁访问某个全局变量，尽量用一个局部变量缓存这个全局变量节省时间

反射优化

反射本质上是调用本地方法，所以执行速度会慢一些，反射优化就是会生成一个新的methodAccessor，把调用本地方法变成调用类的静态方法

JMM（Java内存模型）

原子性

代码临界区的并发问题，如果很多线程同时想访问临界区代码，只能有一个线程可以访问临界区的代码，直到临界区代码访问结束

通过加锁可以保证原子性

可见性

线程内有自己的本地内存（主存的数据copy），JVM有一份共享内存（主存），线程获取变量的值只会从自己的本地内存获取

volatile关键字可以保证线程对共享变量的修改对其他线程可见，也就是线程对这个共享变量的获取要先从主存读取到本地内存才能使用

有序性

指令重排导致代码的执行顺序和预期不一致，最后导致结果有问题

volatile关键字可以禁止指令重排，保证有序性
